=== OBSERVED SYMPTOM ===
4-5 seconds of silence after customer speaks before AI Agent responds.

=== ROOT CAUSE ANALYSIS ===

1. WHY Lex fulfillmentUpdates FILLER DOES NOT WORK
   ------------------------------------------------
   Config: AmazonQinConnect intent fulfillmentUpdatesSpecification.startResponse
     - message: "Sure, one moment please."
     - delayInSeconds: 1
     - active: True
   
   BUT: fulfillmentCodeHook.enabled = False  ← BLOCKING CONDITION
   
   The fulfillmentUpdatesSpecification (startResponse filler) ONLY fires when
   fulfillmentCodeHook.enabled = True AND a Lambda is backing the fulfillment.
   
   For AMAZON.QInConnectIntent, Q in Connect handles fulfillment internally.
   The API does NOT allow setting fulfillmentCodeHook.enabled = True for this
   intent type (returns ValidationException if attempted).
   
   CONCLUSION: Lex filler words are architecturally impossible for this intent type.

1b. WHY Lex initialResponseSetting ALSO DOES NOT WORK
    --------------------------------------------------
    Attempted: add initialResponseSetting to AmazonQinConnect so Lex would speak
    before Q in Connect orchestration starts processing the turn.

    Result: the UpdateIntent call succeeds, but the subsequent bot-locale build
    fails with the service error:
       "Q In Connect intent 'AmazonQinConnect' contains initial response
         configurations. Initial response configurations are not allowed in
         Q In Connect intent."

    CONCLUSION: AmazonQinConnect does not support Lex initialResponseSetting.
    This closes off both Lex filler mechanisms:
       - fulfillmentUpdatesSpecification: blocked by fulfillmentCodeHook.enabled=False
       - initialResponseSetting: blocked by Q In Connect build validation

2. WHY THE AI AGENT PROMPT PRE-TOOL ACKNOWLEDGMENT SHOULD WORK (but may not)
   --------------------------------------------------------------------------
   Prompt v3 contains:
     "CRITICAL FOR VOICE (Nova Sonic streaming): You MUST output a short spoken
      acknowledgment in a <message> block BEFORE invoking ANY tool."
   
   With Nova Sonic speech-to-speech streaming, IF the AI model outputs tokens
   for the acknowledgment before calling a tool, Nova Sonic will stream that
   audio immediately. This is the intended fix.
   
   POTENTIAL ISSUE: Claude Haiku may be ignoring/reordering the instruction
   and calling the tool before generating the acknowledgment, especially for
   simple/short queries where it determines a tool call is the first action.

   This is NOT a flow/config issue - it is LLM + tool call processing latency.

4. WHY CONNECT FLOW CANNOT INSERT A PER-TURN ACKNOWLEDGMENT HERE
    --------------------------------------------------------------
    The active flow uses ConnectParticipantWithLexBot with:
       - Text: the initial greeting spoken once when the bot session starts
       - LexSessionAttributes: audio timing and q-in-connect agent ARN wiring

    The flow regains control only after the Lex/Q in Connect session resolves
    and exits back to the next flow branch. There is no flow action between an
    individual caller utterance and the corresponding AmazonQinConnect tool turn.

    CONCLUSION: the current Connect flow can guarantee the opening greeting, but
    it cannot inject a per-turn acknowledgment before orchestration starts work.
